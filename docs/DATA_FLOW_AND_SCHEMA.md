# Soft-Cases — Data Flow and Schema

**Date:** 10 Mar 2026  
**Service:** `soft-cases`  
**Scope:** standalone Citation Intelligence backend and frontend

---

## 1) System Data Flow

```
Frontend (Next.js)
  ├─ Screen 1: create run
  │    POST /api/v1/runs
  │
  ├─ Screen 2: poll run + jobs
  │    GET /api/v1/runs/{id}
  │    GET /api/v1/runs/{id}/jobs
  │
  └─ Screen 3: fetch results/export
       GET /api/v1/runs/{id}/results
       GET /api/v1/runs/{id}/export/csv
       GET /api/v1/runs/{id}/export/json

FastAPI Router
  └─ Background task: run_pipeline(run_id)
       1. Parse cluster + synthesize queries
       2. Discover (Perplexity, Semantic Scholar, arXiv)
       3. Deduplicate + classify
       4. Score + apply relevance filters
       5. Persist scored set + finalize run

PostgreSQL
  ├─ ci_runs (run state machine + totals)
  ├─ ci_query_jobs (subtopic x source jobs)
  ├─ ci_raw_results (discovered candidates)
  ├─ ci_scored_results (final ranked rows for UI/export)
  └─ ci_clusters (saved cluster configs)
```

---

## 2) Run State Machine

`queued -> synthesising -> discovering -> deduplicating -> scoring -> complete`

Failure path: any non-terminal state can move to `failed`.

Crash recovery: at startup, stale runs older than 30 minutes in non-terminal states are marked `failed` with:
- `error_message="Server restarted during pipeline execution"`

---

## 3) Logical Pipeline Model

### Stage A: Cluster parsing and query synthesis
- Input: `cluster_config` JSON
- Output: parsed hierarchy and per-subtopic queries
- Persist: `ci_query_jobs` rows created with `queued` status

### Stage B: Source discovery
- Sources: `perplexity`, `semantic_scholar`, `arxiv`
- Persist: each discovered item saved to `ci_raw_results`
- Job tracking: each source/subtopic job updated in `ci_query_jobs`

### Stage C: Dedup + classification
- Identity keys (priority): DOI -> arXiv id -> URL
- Classifier sets content type and source tier metadata
- Canonical rows retained; duplicates marked at raw layer

### Stage D: Scoring + filter gates
- Scorer computes relevance and authority signals
- User filter gate applied: `filter_config.min_topical_relevance`
- Exclusion rows are retained with `excluded=true` + reason

### Stage E: Finalize run
- `ci_runs` receives totals and status `complete`
- Results endpoint serves only non-excluded rows by default

## 6) API Contract Notes

### `POST /clusters/generate` (novice flow)
- Input: `{ "topic": "..." }`
- Provider: OpenAI chat completions (`OPENAI_MODEL`, default `gpt-5.3`)
- Output: strict `cluster_config` JSON matching the same schema as manual editor input
- Validation: generated output is re-validated with the same parser used by `POST /runs`
- Failure modes:
  - `503` when `OPENAI_API_KEY` is not configured
  - `422` when provider output cannot be parsed/validated
  - `502` for provider HTTP errors

### `GET /runs/{id}/results`
- Uses a concrete Pydantic response model (`ScoredResultResponse`)
- Serializes SQLAlchemy rows using `from_attributes=True`
- Prevents ORM internal field leaks and JSON serialization failures

### Pagination semantics
- Default `per_page=50`
- Frontend next-page logic should key off raw page size, not displayed row count after client-side hides

### Export semantics
- CSV/JSON exports include excluded rows for complete audit trace
- UI table endpoint excludes excluded rows by default

## 7) Operational Constraints

- `DATABASE_URL` must point to Railway Postgres internal host in production
- `ALLOWED_ORIGINS` must contain fully qualified frontend URL including `https://`
- Frontend standalone server must bind `HOSTNAME=0.0.0.0` in Railway

The current schema is intentionally prefixed and isolated so it can be merged into ACL with minimal table conflicts. The ACL integration gate (Step 4/5) needs:

1. Live ACL DB inspection for `claim_sets` existence
2. Decision on claim parent model (`case` vs `claim_set`)
3. Migration sequencing for CI tables in ACL Alembic chain
